DES, or the Data Encryption Standard, was an older block cipher published by NIST with a very small keyspace (and, as a result, is deprecated). The modern solution was to just perform the DES encryption/decryption process multiple times with several different keys, thus effectively increasing the key space. This is where 3DES comes from. We now discuss block cipher operation modes, which dictate how we actually use block ciphers.

Operation modes typically enhance the effect of a cryptographic algorithm or make it easier to adapt the algorithm for a particular application. Five modes of operation have been defined by NIST, which are intended for use with any symmetric block cipher, including 3DES and AES. These are intended to cover a wide varety of applications of encryption for which a block cipher could be used.

These modes are:

  • electronic codebook (ECB)
  • cipher block chaining (CBC)
  • cipher feedback (CFB)
  • output feedback (OFB)
  • counter (CTR)

Here's an overview of those:

Some measures of quality for each of the cipher modes include:

  • overhead
  • error recovery
  • error propagation
  • diffusion
  • security

With electronic codebook mode, each block is encrypted independently using the same key. This is typically used in the secure transmission of single values (like an encryption key), as the transmission of larger plaintexts can reveal patterns in the underlying plaintext, which is insecure. But the advantage is that because all the blocks are encrypted independently, you can parallelize this.

In cipher block chaining, the output ciphertext of the previous block is XOR'd with the plaintext of the following block before being passed into the encryption function (with the same key); the very first plaintext is XOR'd with an initialization vector, IV, which should be kept private. This is more secure, but the result is that each block generally has to be processed in series.

Note that errors in any particular ciphertext or plaintext "cascade" down to the next iteration. In the case of the plaintext, an error in the plaintext propagates to successive iterations of encryption. But in the case of the ciphertext, an error in a particular ciphertext block only affects the decryption of the next ciphertext block, and no further.

We now discussion stream cipher modes. For AES, DES, or any block cipher, encryption is performed on a block of bits; for DES, this is 64 bits, and for AES, this is 128 bits.

Three modes make it possible to convert a block cipher into a stream cipher - these are cipher feedback (CFB), output feedback (OFB), and counter (CTR) mode. The result is that you can encrypt less data at a time, instead of requiring a full block.

In cipher feedback mode (CFB), you create a system in which the key XOR'd against the plaintext is dynamically generated from passing the same key and IV/ciphertexts through an encryption function as shown, yielding bits equal in length to each (smaller) block of the plaintext:


Again, errors propagate indefinitely here in encryption, but only up to the following plaintext "block" in decrpytion.

In output feedback mode (OFB), the nonce (much like an IV, basically a secret number) is used as the initial input to the encryption function, along with the secret key. The immediate output of the encryption function is used as both the next input and as the effective XOR key.

The advantage with this is that you can precalculate outputs of the encryption function in advance, as it is not dependent on the inputs at all.


Note that unlike the other modes, an error in the plaintext does not propagate to successive ciphertexts, as the effective "key" in the XOR function is independent of the plaintext. Errors in the ciphertext do not propagate for the same reason.

Something of note is that the same encryption function is used for both overall encryption and decryption, as it's being used to generate the "keys" for encryption; in decryption, you want that same key to decrypt, so you naturally will need to use the same encryption function again.

Finally, we have counter (CTR) mode, in which the inputs to the encryption function is a well-defined counter for each stage and the same secret key. The output of the encryption function is again used as the effective XOR key with the plaintext or ciphertext.


The advantage of this is that each block can be processed independently, allowing you to parallelize the operations. You can precalculate the keys in advance given that you know what the future values of the counter will be. And unlike CBC mode, assuming that the counters and encryption function are secure, it can be proven to be secure.

Notably, it is both hardware and software efficient. As mentioned earlier, you can preprocess future "keys"; and because everything is independent, you can randomly decrypt or encrypt parts of a message at will.

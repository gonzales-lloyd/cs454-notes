Digital signatures
...

Message authentication protects two parties who exchange messages against any third party. That is to say, a third party can't mess with the exchange of messages between those two parties.

However, it does not necessarily protect the two parties against each other; anybody can claim that they're the other sender, impersonating the identity of the other sender. So while the messages are secure against third parties, you might not even be communicating with who you think you are. For example, suppose you want to visit a website you've never visited before. How are you going to ensure that you're actually visiting the website you're typing in, and not some impostor?

This introduces digital signatures, which make it possible for the sender to assert they were the sender of the message (and for the recipient to verify the same). To do so, a simple methodology is as follows:

  • The sender calculates a cryptographic hash of their entire message and encrypts it with their private key. The encrypted hash (the signature) is appended to the message.
  • The recipient hashes the message without the signature. Depending on the algorithm, the hash, signature, and public key are used in some way to verify that the hash of the message is what's expected.

In theory, this ensures that any messages that can be verified with the sender's public key were written by that sender, since they are the only one that should hold the private key. Contrast the same procedure with symmetric keys instead; this makes repudiation possible, as anybody who has access to the key (which may be everybody in a system) can claim that they didn't write a message or pose as another person. However, the private key should be uniquely tied to one entity.

This system makes it possible to minimize damages. For example, suppose that a private key is leaked. If the holder of the key notifies all of its recipients that its key has been compromised at a particular date (i.e. the key is revoked or invalidated), then recipients can protect themselves against any messages signed with the leaked public key after that date.

While we won't talk about certificates this class - which tie a public key to a particular organization - this brings us into the idea of chains of trust, in which certain organizations (certificate authority) can vouch for the validity of the certificates held by multiple other organizations. If your browser trusts a particular certificate authority, then any certificates signed by that certificate authority is considered trusted.

Note

The alternative system in which we don't have a chain of trust is where every organization has their own certificates, and then net result is that you have to trust each organization individually. Obviously, this isn't a great system, which is why we trust certificate authorities to do their job in verifying that an organization actually does hold a particular public key, and that the private keys held by certificate authorities used to sign certificates aren't leaked.

Some properties of digital signatures include:

  • it must verify the author, date, and time of the signature
  • it must authenticate the contents at the time of the signature
  • it must be verifiable by third parties to resolve disputes

Some attacks against digital signatures include (in approximately increasing order of knowledge):

  • a key-only attack, in which C only knows A's public key
  • a known message attack, in which C is given access to a set of messages and their signatures
  • a generic chosen message attack, in which C chooses a list of messages before attempting to break A's signature scheme, independent of A's public key. C then obtains from A valid signatures for the chosen messages.
  • a directed chosen message attack, which is similar to the generic chosen message attack except that the list of messages to be signed are chosen after C knows A's public key but before any signatures are seen
  • an adaptive chosen message attack, in which C may request from A signatures of messages that depend on previously obtained message-signature pairs

Ultimately, the goal of attacking digital signatures is to make it possible to forge messages (or break other things depended on the associated private key altogether). In decreasing level of scope, we have:

  • a total break, in which C determines A's private key
  • universal forgery, in which C finds an efficient way to construct valid signatures on arbitrary messages
  • selective forgery, in which C is able to forge a signature for a particular message chosen by C, but not every possible message (for example, perhaps you're only able to forge messages of a certain length or having a particular quality).
  • existential forgery, in which C forges a signature for at least one message that C has no control over

There are various requirements of digital signatures:

  • The signature must depend on the message being signed
  • The signature must use some information unique to the sender to prevent both forgery and denial (which is where private keys come into play)
  • It must be relatively easy to produce, recognize/isolate, and verify signatures
  • It must be computationally infeasible to forge a signature by either constructing a new message for an existing signature or by constructing a new signature for a given message
  • It must be practical to retain a copy of the digital signature in storage (i.e. the signature must be small)

This brings us to direct digital signatures, which are signature schemes that only involve the communicating parties, rather than parties of higher trust (such as certificate authorities). In turn, we assume that each recipient knows the public key of the sender.

In this system, confidentiality can be provided by encrypting the entire message, along with the signature, with a shared secret key. When a dispute occurs, a third party must view the message and verify the signature.

Sidenote

The signature should be applied first, then the whole thing should be encrypted. This is because if the message is already encrypted at signing time, it's not strictly possible to verify that the message hasn't changed from what's expected. Additionally, it's not possible for a third party to verify the original contents of the plaintext message (to resolve a dispute) without the decryption key, which may be undesirable.

The validity of the scheme depends on the security of the sender's private key. One possible way of exploiting this system is by claiming that the private key was falsely lost or stolen, allowing senders to deny sending messages after the fact. One way to weaken this is to require that every signed message include a timestamp and that compromised keys be reported in a prompt manner (typically to some central authority).

Unlike ElGamal, it does not require that the entire message be available before modular exponentiation is perfomed, therefore minimizing the message-dependent amount of computation required to generate a signature. Digital Signature Algorithm
...

NIST published the Digital Signature Algorithm in FIPS 186, which uses SHA; the latest version FIPS 186-3 also incorporates digital signature algorithms based on RSA and ECC.

In DSA, only digital signatures can be used. It cannot be used for encryption or key exchange, even though it operates on a public-key system.

RSA-PSS
...

There is also RSA-PSS, known as the RSA Probabilistic Signature Scheme. As we know, RSA is deterministic - you get the same numbers from the same inputs each time. However, RSA-PSS is probabilistic, in that the same inputs may not yield the same output each time.

For all schemes developed prior to PSS, it has not been possible to develop a proof that the signature scheme is as secure as the underlying RSA cryptographic routines. Unlike other RSA-based schemes, RSA-PSS introduces a randomization process that enables the security of the method to be shown to be closely related to RSA itself.

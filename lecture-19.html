We now move into cryptographic hash functions.

A hash function accepts a variable-length block of data of as an input and produces a fixed-size hash value by calculating .

Generally speaking, the goal of a (cryptographic) hash is data integrity, such that changing a single bit in the input causes a completely different output and it is infeasible to find some target output without brute forcing .

Cryptographic hash functions have the properties that it is computationally infeasible to find either:

  • an input that maps to a target hash result (the one-way property)
  • two data objects that map to the same hash result (the collision-free property)

Naturally, by having an infinite domain map to a finite range (as constrained by the fixed-size hash value), you're going to have collisions. In practice, we avoid this generally by making the range large enough so that collisions are unlikely.

Broadly speaking, cryptographic hash functions accept a message or data block of variable length, where the length may be defined as bits. Typically, a hash function may apply padding based on so that the message is aligned to some byte boundary.

The problem with this is that the hash itself is not protected, so it is trivial to change the contents of the data, recalculate the hash, and forge a "real" message in a man-in-the-middle attack:

In turn, the issue is that we're not authenticating the hash itself. So how do we fix this? There are several options for message authentication.

Remember when we talked about signatures under public-key cryptosystems, a message could be authenticated based on the fact that if a message is encrypted with a private key, anybody with the public key can decrypt the message. But since only one person should (in theory) have the private key, this guarantees that the original message - though accessible by everybody - was written by the claimed sender, who holds the private key.

Similarly, when confidentiality is not required, we can apply a similar idea to "protecting" or authenticating the hash at the end of the message. After generating the message and appending the hash to the end, the sender can simply encrypt the message with their private key. Although anybody can trivially decrypt the message, it is difficult to change the message, change the hash, and then generate a valid ciphertext message since the attacker does not know what the sender's private key is.

There are several possible ways to actually implement this. Some options include:

  • Encrypting the whole message with the public key
  • Encrypting just the hash itself with the public key, so that the hash is protected
  • Using a preshared secret key (or value) that is not sent with the message, but the attached hash to the message is equal to the hash of the message concatenated to the shared secret (so that is required to correctly generate and verify the hash)
  • A combination of the two, where the preshared secret key is again used to protect the hash itself, but the entire message and the protected hash is additionally encrypted using a public-key cryptosystem

Naturally, encryption is not free, and each of these have variable costs.

A more specific application of these approaches is through using Message Authentication Codes, or MACs, which are also known as a keyed hash function. It is typically used between two parties that share a secret key to authenticate information exchanged between those parties. It takes a secret key and a data block that is associated with the protected message; that is to say, it is impossible to alter the associated MAC value (correctly) without knowledge of the secret key. Changing the secret key changes the output of the MAC, as does changing the message.

Digital signatures are similar to that to how MAC works. Here, the hash value of a message is encrypted with a user's private key; anybody who knows the public key can verify the integrity of the message, but only somebody who has the private key can alter the message. Digital signatures can be used for more than just message authentication, which we'll explore later on.

Various other uses of hash functions include:

  • Password storage, where the calculated hash of a plaintext password is what's stored in a database (as opposed to the plaintext password itself)
  • Intrusion and virus detection, where an unexpected change of a file's hash indicates an attacker
  • Pseudorandom functions (and random number generators), such as in the generation of symmetric keys

In a brute-force attack, we simply pick values at random until a collision occurs. Unlike encryption, we aren't methodically exhausting the key space; we may have an arbitrarily large domain to search, and some of our inputs might yield the same outputs (but not the one we want). It's much like rolling a die until you get a particular value; if you want to roll a 6, it takes you 6 tries on average to do so.

(In contrast, for brute-forcing a keyspace, it should take you searching half the keyspace on average. If you had a bag of values from 1-6, and took values out without replacement, it'll take you 3 pulls on average.)

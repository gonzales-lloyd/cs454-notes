But how is a 16-byte key expanded into 44 four-byte words, or 176 total bytes? We need 11 16-byte keys for all of our rounds and the additional transformation, so where does this come from?

But what does do? Given an input word :

  • Rotate the bytes left, yielding
  • Pass each byte in through the same S-box as in the round operations, yielding
  • XOR the result with the round constant , which is defined in the algorithm as a predefined 4-byte word each round
  • Return the result.

The round constant is defined in Section 5.2 in the NIST paper; for AES-128, which has 10 rounds, each round uses a unique constant. For AES-192, the first 8 keys are used, then the first four; for AES-256, the first 7 keys are used twice.

The expansion key algorithm was designed to be resistant to knowncryptanalytic attacks. The use of the round constant in key expansion eliminates the symmetry between the ways in which round keys are generated in different rounds, with various specific criteria used in designing the key expansion algorithm.

Now, an interesting thing is that although AES is symmetric in that both decryption and encryption require the same key, the decryption cipher is not identical to the encryption cipher. The sequence of transformations does differ, but the form of the key schedules is the same.

To bring the decryption structure in line with the encrpytion structure, this means that we need to do:

  • interchange the first two stages of the decryption round
  • interchange the second two stages of the decryption round

The first two stages are the byte substitutions and the row shifting. Since these change the byte contents and the byte sequence independently, these operations (and their inverses, for decryption) can be applied in either order to achieve the same result.

Similarly, the column mixing and addition of the round key do not alter the sequence of the bytes in the state. If we view the key as a sequence of words, then both operate one column at a time, such that the operations are linear with respect to the column input.

So what's the point of all this? Our goal is to use as much of the encryption circuit as possible to implement the decryption circuit. So instead of:

  • substitute bytes through S-box
  • shift rows
  • mix columns
  • add round key

we do the following:

  • inverse S-box substitution
  • inverse row shift
  • inverse column mix
  • add round key, inversing the columns of the round key for all decrpytion rounds past the first

The exact same hardware (with different configuration) can then be used to implement both the encryption and decryption process of AES. Overall, AES can be implemented very efficiently on 8-bit processors; all the individual round operations are relatively simple and require relatively little memory. Designers believe the efficiency of the implementation contributed to Rjindael's selection as the AES cipher.

From a security standpoint, brute force attacks are simply not practical. There is also no known analytical attack that is significantly faster than brute force.

That said, side channel attacks have been published, which attack the implementation of the algorithm rather than the underlying algorithm. This includes cache-timing attacks and power analysis.

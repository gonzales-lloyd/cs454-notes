Naturally, there are many different standardized hashing algorithms that are used today. We'll focus on SHA-2 and SHA-3 today; SHA-1 and SHA-2 are very similar in structure and algorithm, but SHA-3 is intentionally completely different from SHA-2.

At a high level, hashing algorithms generally accept a variable-length message split into several blocks. Several compression functions are chained to each other, such that the input to each function is a chaining variable (the output of the prior function or the IV, in the case of the first input to ) in addition to each block of in series. If the length field is included in the input itself, and if the compression function is collision resistant, then the hash function comprised of a series of these compression function is itself collision resistant. In turn, the problem of designing a secure hash function reduces to that of designing a collision-resistant compression function .

SHA was first published as FIPS 180 in 1993, becoming SHA-1 in 1995. It is largely based on MD4, producing 160-bit hash values. In 2002, NIST produced a revised version of the standard generating three distinct output digest lengths of 256, 384, and 512 bits. Collectively, this is known as SHA-2 (but is typically seen as SHA-256, SHA-512, etc.). You can see a general overview of this below:

Note that the word size is directly correlated to the architecture of the computers they are intended to be run on (i.e. SHA-256 would be for 32-bit architectures, while SHA-384 and up would be for 64-bit architectures). For the descriptions and algorithms that follow, assume that SHA-512 is used; that is, 1024-bit blocks (comprised of 64-bit words) producing a 512-bit digest. The absolute maximum length of an input message to SHA-512 is bits.

At a high level, SHA operates as follows:

  • Append padding bits as needed.
  • Append the length of the padded message to the end of the message.
  • Initialize the hash buffer.
  • Process the message in a series of 1024-bit blocks.
  • Output the result of the final compression function.

Recall that SHA-2 was directly derived from SHA-1, but with longer lengths for everything (thus accounting for improvements in computational power). This is a cause for concern; if a direct vulnerability were to be discovered in SHA-2, it would take years to find a suitable replacement for hashing. (And in 2017, researchers at Google showed that a collision attack on SHA-1 could be done in a reasonable amount of time.)

In turn, NIST announced a competition in 2007 for the SHA-3 function, which was intended to be completely different in design from SHA-2 and was intended to complement SHA-2 as an approved standard for various applications.

SHA-3 (known as Keccak prior to its standardization) uses a scheme known as a "sponge construction", which takes an input message nad partitions it into fixed-size blocks. Each block is processed in series with the output of each iteration fed into the next iteration, finally producing some output block. The sponge function is broadly defined by three parameters:

  • , the internal function used to process each input block
  • , the size in bits of the input blocks, called the bitrate
  • , the padding algorithm.

For SHA-3, two forms of padding are available.

  • Simple padding is where the padded message length is a multiple of the block length, where the padding is all zero bit except for the first, which is a 1.
  • Multirate padding is where padding is added so that the message length is again a multiple of the block length, except that the first and last padding bits are 1s. This is secure even if the rate is changed for a given , and is why the corresponding FIPS uses multirate padding.

SHA-3 has several options for the message digest side; notably, there is no maximum on message size. Naturally, as digest size increases, the collision and second preimage resistance increases (in theory).

Within a single iteration function , 24 rounds are performed in what are known as "step functions." The functions are performed as shown here:

The first step function is the theta step function, in which the new value of each bit in each word depends on its current value and on one bit in each word of the preceding and succeeding columns. In practice, we do each bit in each lane (recall, a cell containing a 64-bit word) in parallel, as is supported by a 64-bit architecture.

At a high level, we're simply summing the bits in each column (for the preceding column, it's the corresponding bit; for the succeeding column, it's the bit immediately after), then XOR'ing the results with the 64-bit word of the cell currently being processed.

Lecture 13

We now move onto asymmetric ciphers, in which we have two asymmetric keys; if you encrypt with one key, you must decrypt with the other key. Deriving the private key from the public key must be computationally infeasible.

Asymmetric keys are two related keys - a public and private key - that are used to perform complementary operations, such as encryption and decryption or signature generation and signature verification.

Public key certificates are digital documents issued and digitally signed by the private key of a certification authority, binding the name of a subscriber to a public key. The certificate indicates that the subscriber identified in the certificate has sole control and access to the corresponding private key.

Public key certificates are part of public key infrastructure (PKI), which is a set of policies, processes, server platforms, software, and workstations that are used for the purpose of administering certificates and asymmetric key pairs. PKI also involves the ability to issue, maintain, and revoke public key certificates.

There are some common misconceptions with asymmetric (public key) encryption:

  • Public key encryption is not necessarily more secure from cryptanalysis than symmetric encryption. The security depends on the length of the asymmetric keys, as well as the computational work involved in breaking a cipher based on these keys.
  • Public key encryption does not make symmetric encryption obsolete; the computational overhead of current public key encryption schemes is high.
  • The distribution of public keys is not necessarily "easier" than the handshakes involved with the distribution of symmetric keys. You still need a protocol and often a central agent for your public keys, which is not simpler nor more efficient than symmetric key distribution.

Fundamentally, public key cryptography aims to solve two problems associated with symmetric encryption:

  • Key distribution: How do we engage in secure communications without needing to trust a key distribution center with your key?
  • Digital signatures: How do we verify that a message came intact from the claimed sender? (That is to say, how do we verify integrity? We'll also deal with authenticity - that it came from a particular sender - later on.)

Whitfield Diffie and Martin Hellman came up with a method that addressed these problems, and was radically different from all previous approaches to cryptography. Broadly speaking, public key cryptography naturally involves a plaintext, ciphertext, and encryption/decryption algorithm. But you also have your private and public keys, which can both be used for either encryption or decryption.

It should be computationally infeasible to determine the decryption key given only knowledge of the cryptographic algorithm and the encryption key.

Here's the general steps of asymmetric cryptography:

  • Each user (Alice, Bob) generates a pair of keys to be used in both encryption and decryption. The private key is never shared.
  • Both users share their public key with everybody, such that other user has access to the other user's public key.
  • If Bob wants to send a confidential message to Alice, Bob encrypts the message using Alice's public key. Bob can then send the message over insecure channels; since Alice is the only one with her private key, she is the only one that can decrypt Bob's message.
    • The same holds in both directions; you can encrypt with the private key and decrypt with the public key, and both Alice and Bob can do this. Encrypting data with the private key is often used in authentication; if Bob encrypts data with his private key, anybody can use his public key to decrypt - so the message isn't secure. But that's not the goal; since only Bob has the private key, the message must have come from Bob.

The important thing to note here is that security can be maintained without providing the key necessary for decryption to everybody who needs to send a message to you. That is to say:

  • In conventional encryption:
    • The same algorithm with the same key is used for encryption and decryption.
    • The sender and receiver must share the algorithm and the key.
    • For security to be maintained, the key must be kept secret; it must also be impossible to decipher a message if the key is kept secret. Knowledge of the algorithm and samples of ciphertext must be insufficient to determine the key.
  • In asymmetric encryption:
    • One algorithm is used for encryption and a related algorithm is used for decryption, both of which use a pair of keys.
    • The sender and receiver must each have one of the matched pair of keys.
    • For security to be maintained, one of the two keys must be kept secret; it must be impossible or impractical to decipher a message encrypted with one key if the other key is kept secret (which in practice, means encryption with public keys should be secure if the private key is kept safe). Knowledge of the algorithm, one key, and samples of ciphertext should be insufficient to determine the other key.

In the context of confidentiality, the sender uses the public key to encrypt a message and send it over an insecure channel. The receiver then uses their private key to decrypt the message. A cryptanalyst in this case has access to the public key and the encrypted message; ideally, they would like to recover the private key and/or the decrypted message .

Similarly, in the context of authentication, the sender uses the private key to encrypt a message and sends it publicly. Any receiver can then use the known public key to decrypt, effectively verifying the message came from the claimed sender. In this case, a cryptanalyst only wants to recover the private key, since the plaintext can easily be recovered.

The fact is that you can have both authentication and confidentiality in the same message. If Bob uses his own private key to encrypt a message, and then uses Alice's public key to encrypt it again, only Alice can use the private key to decrypt the message to the point where she can then use Bob's public key to verify that it came from Bob. In practice, we typically use dedicated systems as opposed to double encryption.

Broadly speaking, asymmetric cryptography has three main uses:

  • Encryption and decryption (where senders encrypt using the recipient's public key)
  • Digital signatures (where senders encrypt using their own private key, allowing the message to be verified as coming from them)
  • Key exchange (where two sides cooperate to exchange a session key, usually a secret key used in faster symmetric encryption protocols)

Not all asymmetric algorithms can be used for all three. For example, RSA and ECC (elliptic curve cryptography) can be used for all three, but the Diffie-Hellman algorithm/system can only be used in key exchange.

Formally speaking, asymmetric algorithms must fulfill the following:

  • It should be computationally easy for a party B to generate a public/private key pair
  • It should be computationally easy for a sender A to generate a ciphertext based on knowledge of the public key
  • It should be computationally easy for the receiver B to decrypt the ciphertext from A, using only the private key
  • It should be computationally difficult for an attacker to determine the private key from the public key
  • It should be computationally difficult for an attacker to recover a message encrypted with the private key given only the public key and the ciphertext
  • The keys can be applied in either order; that is, both keys must be valid for either encryption or decryption

For an asymmetric algorithm to even be possible, you need a trapdoor (one-way) function. That is to say, it should map a domain to a range such that every function value has a unique inverse; calculating the result of applying the function should be easy, but calculating the inverse of a result should be infeasible.

  • is difficult if only is known

Asymmetric encryption is still vulnerable to brute force attacks (and naturally, the counter is to use really large keys). The key size must still be small enough for practical encryption and decryption; modern key sizes result in encryption/decryption speeds that are simply too slow for general use, when compared to symmetric algorithms. In turn, public key encryption is only used in key management and signatures in practice.

To date, there is no mathematical proof that it is infeasible to recover the private key given the public key for a particular algorithm that generates public/private key pair. To the best of our knowledge, there isn't a way to do so, but that doesn't mean that no such method exists.

Finally, there is a probable-message attack, where you encrypt all possible plaintext messages and try to find a match between a ciphertext and your calculated ciphertexts.
